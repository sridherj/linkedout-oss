# Decision: Environment & Configuration Design

**Spike:** 0G  
**Date:** 2026-04-07  
**Status:** Approved by SJ (2026-04-07)  
**Author:** Claude (taskos-detailed-plan agent), refined interactively with SJ

---

## Context

LinkedOut OSS needs a clean, OSS-friendly configuration system that replaces the current private-repo patterns (`.env.local`/`.env.test`/`.env.prod` per-environment files, scattered `os.getenv()` calls, hardcoded constants). The system must support:

- Self-hosted single-user installs (primary use case)
- No Docker, no cloud services required
- Secrets kept out of the repo and out of env files where possible
- Human-readable config that non-engineers can edit
- Env var overrides for CI, automation, and power users

---

## Decision Summary

A **three-layer config hierarchy** with env vars > YAML config > code defaults:

1. **`~/linkedout-data/config/config.yaml`** — human-readable YAML for all user-facing settings
2. **`~/linkedout-data/config/secrets.yaml`** — API keys and credentials, `chmod 600`, separate from config
3. **Environment variables** — override any YAML value; used by CI, scripts, and power users

Validated at startup by **pydantic-settings** with clear error messages for missing required values.

---

## Config Hierarchy (Precedence: highest wins)

```
Environment variable          # LINKEDOUT_EMBEDDING_PROVIDER=local
  ↓ overrides
~/linkedout-data/config/config.yaml      # embedding_provider: openai
  ↓ overrides
~/linkedout-data/config/secrets.yaml     # openai_api_key: sk-...
  ↓ overrides
Code defaults                 # embedding_provider = "openai"
```

**Resolution rule:** For any setting, the first defined value wins in the order above. Environment variables always win.

---

## Directory Layout

**Single unified directory** — one place to know about, one dir to delete for clean slate, one root for diagnostics to scan.

```
~/linkedout-data/                  # LINKEDOUT_DATA_DIR
├── config/
│   ├── config.yaml                # User settings (all non-secret config, including DATABASE_URL)
│   ├── secrets.yaml               # API keys only — chmod 600 (OPENAI_API_KEY, APIFY_API_KEY, LANGFUSE_*)
│   ├── agent-context.env          # Generated by setup — DB URL + tenant/BU/user IDs for Claude skills
│   └── .last-setup-version        # Tracks last setup version run
├── db/                            # PostgreSQL data (or connection config pointer)
├── crawled/                       # LinkedIn profile data from extension
├── uploads/                       # User-uploaded CSVs, VCFs
├── logs/                          # All application logs
├── queries/                       # Query history (YYYY-MM-DD.jsonl)
├── reports/                       # Readiness reports, operation artifacts
├── metrics/                       # Local metrics (append-only JSONL)
├── seed/                          # Downloaded seed data (SQLite files)
└── state/                         # Embedding progress, sync state
```

**Configurable** via `LINKEDOUT_DATA_DIR` env var. Default: `~/linkedout-data`.

**Clean slate:** `rm -rf ~/linkedout-data/` removes everything — config, secrets, data, logs. Re-run `/linkedout-setup` to rebuild.

**Agent context:** `~/linkedout-data/config/agent-context.env` is auto-generated by setup and contains:
- `DATABASE_URL` (unredacted — single-user localhost, password is not a real security boundary)
- `LINKEDOUT_TENANT_ID`, `LINKEDOUT_BU_ID`, `LINKEDOUT_USER_ID` (system defaults)
Skills/CLAUDE.md reference this file so Claude doesn't guess at DB connection or RLS context.

---

## Naming Convention: `LINKEDOUT_` Prefix

All LinkedOut-specific env vars use the `LINKEDOUT_` prefix. Industry-standard vars (`DATABASE_URL`, `OPENAI_API_KEY`, etc.) keep their standard names so users don't need to learn non-standard mappings.

**Rationale:** If a user already has `OPENAI_API_KEY` set for other tools, LinkedOut should just work. If they need a LinkedOut-specific override, they can use `LINKEDOUT_OPENAI_API_KEY` (future, not v1).

### YAML-to-env mapping convention

YAML keys use `snake_case`. The corresponding env var is `LINKEDOUT_` + `UPPER_SNAKE_CASE`:
- `embedding_provider` ↔ `LINKEDOUT_EMBEDDING_PROVIDER`
- `log_level` ↔ `LINKEDOUT_LOG_LEVEL`

**Exceptions** (industry-standard names, no prefix):
- `database_url` ↔ `DATABASE_URL`
- `openai_api_key` ↔ `OPENAI_API_KEY`
- `apify_api_key` ↔ `APIFY_API_KEY`
- `langfuse_*` ↔ `LANGFUSE_*`

## `~/linkedout-data/config/secrets.yaml` Format

```yaml
# LinkedOut Secrets — chmod 600
# These values can also be set as environment variables.

# ── API Keys ─────────────────────────────────────────────
openai_api_key: sk-...

# ── Optional ─────────────────────────────────────────────
# apify_api_key: apify_api_...
# langfuse_public_key: pk-...
# langfuse_secret_key: sk-...
```

**Security requirements:**
- Created by `/linkedout-setup` with `chmod 600` (owner read/write only)
- Setup skill warns if permissions are too open (like SSH does for `~/.ssh/`)
- Never read by any tool except LinkedOut itself
- `.gitignore` patterns prevent accidental commits: `secrets.yaml`, `*.secret`, `*.key`

## Pydantic-Settings Implementation

### Loading order in code

```python
from pydantic_settings import BaseSettings, SettingsConfigDict
from pydantic import Field
import yaml, os

class LinkedOutSettings(BaseSettings):
    """
    Resolution order (highest priority wins):
    1. Environment variables
    2. .env file (if present)
    3. secrets.yaml
    4. config.yaml
    5. Code defaults
    """
    model_config = SettingsConfigDict(
        env_prefix='LINKEDOUT_',
        env_file='.env',
        env_file_encoding='utf-8',
        extra='ignore',
        case_sensitive=False,
    )

    # Core
    database_url: str = Field(
        default='postgresql://linkedout:@localhost:5432/linkedout',
        validation_alias='DATABASE_URL',  # no prefix for standard var
    )
    data_dir: str = Field(default='~/linkedout-data')
    config_dir: str = Field(default='~/.linkedout')
    environment: str = Field(default='local')
    debug: bool = Field(default=False)

    # Server
    backend_host: str = Field(default='localhost')
    backend_port: int = Field(default=8001)
    backend_url: str = Field(default='')  # computed if empty
    cors_origins: str = Field(default='')

    # Embeddings
    embedding_provider: str = Field(default='openai')
    embedding_model: str = Field(default='')

    # LLM
    llm_provider: str = Field(default='openai')
    llm_model: str = Field(default='')

    # API Keys (standard names, no prefix)
    openai_api_key: str | None = Field(
        default=None, validation_alias='OPENAI_API_KEY'
    )
    apify_api_key: str | None = Field(
        default=None, validation_alias='APIFY_API_KEY'
    )

    # Logging
    log_level: str = Field(default='INFO')
    log_format: str = Field(default='human')
    log_dir: str = Field(default='')  # defaults to data_dir/logs

    # Langfuse (standard names, no prefix)
    langfuse_enabled: bool = Field(
        default=False, validation_alias='LANGFUSE_ENABLED'
    )

    # Extension tuning
    rate_limit_hourly: int = Field(default=30)
    rate_limit_daily: int = Field(default=150)
    staleness_days: int = Field(default=30)
    enrichment_cache_ttl_days: int = Field(default=90)

    @classmethod
    def settings_customise_sources(cls, settings_cls, **kwargs):
        """Custom source order: env > .env > secrets.yaml > config.yaml > defaults."""
        return (
            kwargs['env_settings'],
            kwargs['dotenv_settings'],
            YamlSecretsSource(settings_cls),
            YamlConfigSource(settings_cls),
            kwargs['init_settings'],
        )
```

### Custom YAML sources (sketch)

```python
class YamlConfigSource:
    def __init__(self, settings_cls):
        config_dir = os.environ.get('LINKEDOUT_DATA_DIR', '~/.linkedout')
        self.path = os.path.expanduser(f'{config_dir}/config.yaml')

    def __call__(self):
        if not os.path.exists(self.path):
            return {}
        with open(self.path) as f:
            return yaml.safe_load(f) or {}

class YamlSecretsSource:
    def __init__(self, settings_cls):
        config_dir = os.environ.get('LINKEDOUT_DATA_DIR', '~/.linkedout')
        self.path = os.path.expanduser(f'{config_dir}/secrets.yaml')

    def __call__(self):
        if not os.path.exists(self.path):
            return {}
        with open(self.path) as f:
            return yaml.safe_load(f) or {}
```

## Extension Configuration

### Build-time (baked into the extension)

The extension is shipped as a pre-built zip. Build-time config via Vite env vars:

```typescript
// wxt.config.ts — resolved at build time
const BACKEND_URL = import.meta.env.VITE_BACKEND_URL || 'http://localhost:8001';
```

Default builds use `http://localhost:8001`. Users who run the backend on a different port must rebuild the extension (or use the options page).

### Runtime (extension storage)

`lib/constants.ts` is refactored to read from `browser.storage.local` with hardcoded fallbacks:

```typescript
// lib/config.ts (replaces hardcoded constants.ts)
export interface ExtensionConfig {
  backendUrl: string;        // default: VITE_BACKEND_URL or http://localhost:8001
  stalenessdays: number;     // default: 30
  hourlyLimit: number;       // default: 30
  dailyLimit: number;        // default: 150
}

const DEFAULTS: ExtensionConfig = {
  backendUrl: import.meta.env.VITE_BACKEND_URL || 'http://localhost:8001',
  stalenessdays: 30,
  hourlyLimit: 30,
  dailyLimit: 150,
};

export async function getConfig(): Promise<ExtensionConfig> {
  const stored = await browser.storage.local.get('linkedout_config');
  return { ...DEFAULTS, ...stored.linkedout_config };
}
```

An **options page** (`entrypoints/options/`) lets users edit these values through the extension UI. This is Phase 12 work.

## Startup Validation

On startup, the config system:

1. **Loads** config.yaml, secrets.yaml, .env, env vars (in precedence order)
2. **Validates** all values via pydantic (types, ranges, required fields)
3. **Expands** `~` in all path values
4. **Creates** data directories if they don't exist (`LINKEDOUT_DATA_DIR` tree)
5. **Warns** if `secrets.yaml` has loose permissions (not `0600`)
6. **Fails fast** with a clear, actionable error if `DATABASE_URL` is missing or malformed

Example startup error:
```
ERROR: LinkedOut configuration invalid.

  DATABASE_URL is required but not set.

  Set it in one of:
    1. ~/linkedout-data/config/config.yaml  →  database_url: postgresql://...
    2. Environment variable       →  export DATABASE_URL=postgresql://...
    3. .env file in project root  →  DATABASE_URL=postgresql://...

  The setup skill generates this automatically: /linkedout-setup
```

## Decisions & Rationale

| # | Decision | Choice | Rationale |
|---|----------|--------|-----------|
| 1 | Config file format | YAML | Human-readable, familiar, no ambiguity with types (unlike .env). Users can comment. |
| 2 | Separate secrets file | Yes (`secrets.yaml`) | Defense in depth — users who share `config.yaml` in dotfiles don't leak keys. `chmod 600` enforced. |
| 3 | `LINKEDOUT_` prefix | Yes, for LinkedOut-specific vars | Avoids collision with other tools. Industry-standard vars (`DATABASE_URL`, `OPENAI_API_KEY`) keep standard names. |
| 4 | Env vars override YAML | Yes | Standard 12-factor pattern. CI and automation need env var overrides. |
| 5 | No per-environment files | Correct — single config.yaml | OSS users run one environment. `LINKEDOUT_ENVIRONMENT` is for test isolation, not user-facing. |
| 6 | pydantic-settings | Yes | Already in use. Custom sources for YAML loading. Type validation, clear errors. |
| 7 | `config.yaml` location | `~/linkedout-data/config/` | Standard XDG-ish pattern. Separate from code (repo deletion doesn't lose config). Configurable via `LINKEDOUT_DATA_DIR`. |
| 8 | Data directory | `~/linkedout-data/` | Visible, human-browsable location. Not hidden (unlike `~/linkedout-data/config/`). Configurable via `LINKEDOUT_DATA_DIR`. |
| 9 | Extension config | Storage API + build-time env | Extensions can't read YAML. `VITE_BACKEND_URL` for build-time, `browser.storage.local` for runtime. Options page in Phase 12. |
| 10 | Firebase auth | Remove for OSS | Single-user local install doesn't need Firebase. `AUTH_ENABLED=false` by default. |
| 11 | Apify key rotation | Single key | OSS = single user. No need for round-robin `APIFY_API_KEY_1..9`. |
| 12 | Default log format | `human` | OSS users read logs in terminal. `json` available for automation. |

## Resolved Decisions (from SJ review)

1. **Unified directory** — All config, data, and logs under `~/linkedout-data/`. No separate `~/.linkedout/` dir. One place to know about, clean slate = delete one dir.

2. **DB password in config.yaml** — `DATABASE_URL` stays in `config.yaml`, not `secrets.yaml`. It's single-user localhost — the password is not a real security boundary. `secrets.yaml` is only for API keys.

3. **No backward compat** — Clean break. This is a fresh OSS project with no existing users. No deprecated env var mappings.

4. **Agent context file** — `~/linkedout-data/config/agent-context.env` generated by setup. Contains `DATABASE_URL` + system tenant/BU/user IDs. Skills/CLAUDE.md reference this file so Claude doesn't guess at DB connection or RLS context — avoids infinite retry loops when Claude can't find credentials or gets zero results from RLS-filtered queries.
