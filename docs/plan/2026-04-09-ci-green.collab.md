## Problem 1: Unit test job hangs indefinitely

**Root cause:** `tests/smoke/test_demo_search_smoke.py` has no pytest marker. The unit test run collects it. The file does `psycopg2.connect(...)` at test time, which blocks forever waiting for a PostgreSQL server that doesn't exist in the unit test runner.

**Evidence:** CI unit test job ran for 6+ hours before we added a timeout.

**Decision:** **B** — Add `--ignore=tests/smoke/` to CI unit test command. File is already in `tests/smoke/` directory. Also add `timeout-minutes: 20` to all CI jobs as a safety net.

## Problems 2+3+4 (consolidated): Test env config via `os.environ` hacking

**Root causes (three intertwined issues):**

1. `backend/conftest.py:126` unconditionally sets `DATABASE_URL`, overwriting CI's value. Integration tests fail with `password authentication failed for user "linkedout"`.

2. `backend/conftest.py:127` sets `LINKEDOUT_EMBEDDING_PROVIDER` (single underscore) — wrong env var name. Pydantic-settings needs `LINKEDOUT_EMBEDDING__PROVIDER` (double underscore, `env_nested_delimiter='__'`). The var is silently ignored, provider defaults to `openai`, integration tests blow up with `OPENAI_API_KEY is not configured`. Also: it's set globally, breaking unit test `test_openai_without_api_key_warns` which expects the `openai` default.

3. `backend/tests/integration/conftest.py:47` sets `LINKEDOUT_ENVIRONMENT=integration_test` at module level. pytest imports ALL conftest.py files during collection — even for unit test runs — so this overwrites the root conftest's `test` value.

**All three are symptoms of the same anti-pattern:** conftest files mutating `os.environ` to configure the application.

**Decision: Use `.env.test` files.**

Conftest stops being a config manager. Instead:

- `backend/.env.test` — loaded by root conftest for unit tests:
  ```
  LINKEDOUT_ENVIRONMENT=test
  DATABASE_URL=postgresql://linkedout:test@localhost:5432/linkedout_test
  ```

- `backend/.env.integration` — loaded by integration conftest for integration tests:
  ```
  LINKEDOUT_ENVIRONMENT=integration_test
  LINKEDOUT_EMBEDDING__PROVIDER=local
  ```

- CI workflow YAML sets env vars in the `env:` block (takes precedence over `.env` files).
- Local dev uses `.env.local` (already gitignored, already loaded).
- Root conftest: `load_dotenv('.env.test', override=False)` — no `os.environ` assignments.
- Integration conftest: `load_dotenv('.env.integration', override=False)` — no module-level `os.environ`.
- `override=False` ensures CI env vars and `.env.local` are never clobbered.

**Files:**
- Create `backend/.env.test`
- Create `backend/.env.integration`
- Modify `backend/conftest.py` — replace `os.environ` lines with `load_dotenv`
- Modify `backend/tests/integration/conftest.py` — replace module-level `os.environ` with `load_dotenv`
- Add `LINKEDOUT_EMBEDDING__PROVIDER: local` and `TENANT_SECRET_ENCRYPTION_KEY` to CI integration job env

## Problems 5+6 (consolidated): SQLite test fixture impedance mismatch

**Root causes:**

- P5: `generate_test_seed.py` creates column `active` (line 47). The OSS Alembic migration `001_baseline.py` creates `is_active`. When `import_seed` inserts from the fixture, the column name doesn't match.

- P6: SQLite stores booleans as INTEGER (0/1). PostgreSQL rejects integer where boolean expected. `import_seed.py` has a `BOOL_COLUMNS` mapping but only lists `startup_tracking.watching` — missing `is_active` on all 6 tables.

**Context:** The demo pipeline already uses pg_dump/pg_restore (`backend/src/linkedout/demo/db_utils.py`) and doesn't have these impedance mismatch issues at all. The question is whether to patch the SQLite fixture or switch the seed format to pg_dump.

**Options:**

| # | Approach | Pros | Cons |
|---|----------|------|------|
| A | Fix the fixture generator: rename `active` → `is_active`, add `is_active` to `BOOL_COLUMNS` for all 6 tables, regenerate fixture | Minimal change. Fixes CI now. | SQLite impedance mismatch remains as a class of bugs — future schema changes can re-introduce similar issues. |
| B | Switch seed data format from SQLite to pg_dump (reuse demo pipeline infrastructure) | Eliminates the entire class of type-mismatch bugs. Aligns with existing demo pipeline pattern. | Larger change — requires rewriting `seed_export.py`, `import_seed.py`, `generate_test_seed.py`, and all tests that reference `.sqlite`. Out of scope for CI green. |
| C | Hybrid: Fix the fixture now (A), then switch to pg_dump as a follow-up | CI green fast. Clean architecture later. | Two rounds of changes. |

**Decision:** **C** — Fix the fixture now to get CI green. The pg_dump switch is a separate initiative that touches the entire seed pipeline (export, download, import, tests). It's the right long-term direction but shouldn't block CI green.

**Immediate fix:**
1. Fix `generate_test_seed.py`: rename `active` → `is_active` in all 6 table schemas
2. Add `is_active` to `BOOL_COLUMNS` in `import_seed.py` for all 6 tables (this IS a product bug — any user importing seed data would hit it)
3. Regenerate `test-seed-core.sqlite` by running the fixed generator

## Problem 9: Session poisoning across integration tests (InFailedSqlTransaction)

**Root cause:** `integration_db_session` is session-scoped (shared across all tests on a pytest-xdist worker). If any test causes a DB error without rollback, the session enters "aborted transaction" state. All subsequent tests on that worker fail with `InFailedSqlTransaction`.

**Primary trigger:** `test_bad_table_returns_hint` executes invalid SQL. The `execute_sql` function uses a savepoint and rolls it back, but something still leaves the outer session dirty.

**Evidence:** After `test_bad_table_returns_hint` passes on gw0, ALL subsequent tests on gw0 fail with `InFailedSqlTransaction`.

**Decision:** Two-part fix:
1. Add `autouse=True` function-scoped fixture in integration conftest that checks for dirty session pre-test and rolls back post-test. Prevents ALL cascading failures.
2. Fix `vector_column_ready` fixture in intelligence conftest — add `session.rollback()` in the `except` block instead of silently swallowing the error.

## Problem 10: RLS tests fail because CI DB user is superuser

**Root cause:** CI postgres container creates `linkedout_test` as the database owner/superuser. PostgreSQL superusers bypass RLS even with `FORCE ROW LEVEL SECURITY`. The RLS tests expect row filtering but get all rows.

**Evidence:** `assert 25 == 20` (user A sees all 25 connections instead of their 20).

**Decision:** Remove superuser from `linkedout_test` in CI after migrations run. No test code changes needed.

**Implementation:** Add one CI YAML step between Alembic migrations and test execution:
```yaml
- name: Remove superuser (so RLS policies apply)
  env:
    DATABASE_URL: postgresql://linkedout_test:linkedout_test@localhost:5432/linkedout_test
  run: psql $DATABASE_URL -c "ALTER ROLE linkedout_test NOSUPERUSER;"
```

**Audit confirms safe:** `linkedout_test` remains database owner and table owner. All integration test operations (CREATE/DROP SCHEMA, CREATE TABLE, ALTER TABLE, CREATE INDEX, RLS policies, TRUNCATE CASCADE, DML) work without superuser. The only superuser-requiring operation (`CREATE EXTENSION vector`) is already handled by Alembic migrations which run before `NOSUPERUSER`.

## Problem 11: Seed pipeline test FK violation

**Root cause:** `_clear_seed_tables()` uses `DELETE FROM {table}` in reverse IMPORT_ORDER. But non-seed tables (like `experience`) have FK references to seed tables (like `company`). DELETE fails with FK violation.

**Evidence:** `foreign key constraint "experience_company_id_fkey" on table "experience"`

**Decision:** Use `TRUNCATE {tables} CASCADE` instead of individual DELETEs. Handles all FK chains in a single statement. Clean and correct for test cleanup.

---

## Problem 12: Extension build workflow missing permissions

**Root cause:** The `extension-build.yml` workflow uses `gh release upload` which needs write access to contents. The default `GITHUB_TOKEN` permissions don't include this.

**Evidence:** 403 error on release asset upload.

**Decision:** Add `permissions: contents: write` to the workflow. Standard practice.

## Follow-up: Switch seed format to pg_dump (separate plan)

The SQLite impedance mismatch (P5, P6) is a symptom of a larger design issue. The demo pipeline already uses pg_dump/pg_restore via `db_utils.py` and doesn't have these problems. Switching the seed pipeline to the same format would:

- Eliminate boolean, array, and timestamp type conversion code in both export and import
- Remove `BOOL_COLUMNS`, `ARRAY_COLUMNS`, and all `_convert_row` logic
- Simplify `import_seed.py` from ~400 lines to ~50 (delegate to pg_restore)
- Align seed and demo pipelines on the same format

This is tracked in the seed data plan and should be addressed there, not here.
