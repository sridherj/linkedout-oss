## Phase Overview

Phase 8 introduces the skill system — LinkedOut's "frontend." Users never interact with CLI commands directly; they invoke `/linkedout` and the skill translates natural language to CLI calls and DB queries. This phase delivers:

1. **Template system** — `SKILL.md.tmpl` files with host-specific variable resolution, following gstack's proven pattern
2. **Host configs** — Per-platform configuration for Claude Code, Codex, and Copilot (frontmatter rules, path rewrites, tool name mappings)
3. **Six skill definitions** — `/linkedout`, `/linkedout-setup`, `/linkedout-upgrade`, `/linkedout-history`, `/linkedout-report`, `/linkedout-setup-report`
4. **Routing** — CLAUDE.md / AGENTS.md with pattern-based dispatch rules
5. **Generation script** — `bin/generate-skills` that renders templates to per-host output
6. **Dev workflow** — `bin/dev-setup`, `bin/dev-teardown`, `linkedout-relink` for contributors

The skill *content* for setup (Phase 9), upgrade (Phase 10), and history/reporting (Phase 11) will be minimal stubs in this phase — the full interactive flows are built in those phases. The infrastructure and the core `/linkedout` query skill are the deliverables here.

## Task Breakdown

### 8A. SKILL.md Template System

**Goal:** Build a template engine that renders `SKILL.md.tmpl` files into host-specific `SKILL.md` files. Follow gstack's pattern: each skill directory contains a `.tmpl` file, and the generation script resolves variables per host.

**Acceptance criteria:**
- Template files use a simple variable syntax (e.g., `{{VARIABLE_NAME}}`)
- Variables are resolved from host config + global context
- Templates can include conditional blocks for host-specific content (e.g., `{{#if claude}}...{{/if}}`)
- Frontmatter is processed per host config rules (strip fields, description limits, allowlist/denylist)
- Path rewrites are applied (e.g., `.claude/skills/` → `.agents/skills/` for Codex)
- Tool name rewrites are applied where needed (e.g., `AskUserQuestion` for Claude, `input` for Codex if applicable)
- Generated files have a header comment: `<!-- Generated by bin/generate-skills — do not edit -->`

**Files to create:**
- `skills/lib/template.py` — Template engine: load `.tmpl`, resolve variables, apply host config
- `skills/lib/config.py` — Host config loader, global context (version, repo URL, etc.)
- `skills/lib/frontmatter.py` — Frontmatter processing (parse YAML frontmatter, apply host rules)

**Design decisions:**
- **Python, not TypeScript** — gstack uses TypeScript because gstack is a TypeScript project. LinkedOut's backend is Python, and the generation script should match the project's primary language. No Node.js dependency for users.
- **Simple templating** — Use Python's `string.Template` or a minimal Jinja2 subset. No heavy template engine. gstack's templates are simple variable substitution + conditionals, not complex logic.
- **Template variables** — Define a clear set: `{{SKILL_NAME}}`, `{{SKILL_DESCRIPTION}}`, `{{VERSION}}`, `{{DATA_DIR}}`, `{{CONFIG_DIR}}`, `{{CLI_PREFIX}}`, `{{HOST_NAME}}`, `{{AGENT_CONTEXT_PATH}}`, `{{DB_SCHEMA_SNIPPET}}`

**Integration points:**
- `docs/decision/env-config-design.md` — `~/linkedout-data/` paths, `agent-context.env` location
- `docs/decision/cli-surface.md` — CLI command names used in skill templates

**Testing:**
- Unit: Template engine resolves variables correctly
- Unit: Frontmatter processing strips/keeps fields per host config
- Unit: Path rewrites transform correctly
- Integration: Generate all skills for all hosts, verify output is valid markdown

**Complexity:** M

### 8B. Host Configs

**Goal:** Define per-host configuration for Claude Code, Codex, and Copilot. These configs tell the generation script how to transform templates for each platform.

**Acceptance criteria:**
- Each host has a config file defining: display name, skill install path, frontmatter rules, path rewrites, tool rewrites
- Claude Code is the primary host — templates are written for it, other hosts are derived via config
- Codex config handles: `.agents/skills/linkedout/` path, frontmatter allowlist mode, description length limit, `$LINKEDOUT_ROOT` env var for paths
- Copilot config handles: `.github/skills/linkedout/` or `.copilot/skills/linkedout/` path, appropriate frontmatter rules

**Files to create:**
- `skills/hosts/claude.yaml` — Claude Code host config
- `skills/hosts/codex.yaml` — Codex host config
- `skills/hosts/copilot.yaml` — Copilot host config

**Host config schema (YAML, matching gstack's structure but in YAML not TypeScript):**
```yaml
name: claude
display_name: Claude Code
skill_install_path: ~/.claude/skills/linkedout
local_skill_path: .claude/skills/linkedout
frontmatter:
  mode: denylist          # denylist = strip listed fields; allowlist = only keep listed
  strip_fields: []
  description_limit: null
path_rewrites: []         # [{from: "...", to: "..."}]
tool_rewrites: {}         # {AskUserQuestion: input}
```

**Integration points:**
- gstack reference: `<reference>/gstack/hosts/` — follow the same config shape
- `docs/decision/2026-04-07-skill-distribution-pattern.md` — SKILL.md standard

**Testing:**
- Unit: Host config loads and validates
- Unit: Each host config produces valid output when applied to a test template

**Complexity:** S

### 8C. `/linkedout` Skill (Core Query Skill)

**Goal:** The flagship skill. Users invoke `/linkedout` with natural language queries about their professional network. The skill routes to psql queries or CLI commands, formats results, and suggests follow-ups.

**Acceptance criteria:**
- Skill template at `skills/linkedout/SKILL.md.tmpl`
- Generated output at `skills/claude-code/linkedout/SKILL.md`, `skills/codex/linkedout/SKILL.md`, `skills/copilot/linkedout/SKILL.md`
- Skill loads `agent-context.env` to get DB credentials and tenant/BU/user IDs
- Skill contains a DB schema reference (key tables, columns, relationships) so Claude can write queries
- Skill includes example queries and SQL patterns for common lookups
- Skill runs `linkedout status --json` as a preamble to check system health
- Skill formats results as tables/summaries
- Skill suggests follow-up queries based on results

**Skill content structure:**
1. **Frontmatter** — name, description, allowed tools (Bash, Read, Grep)
2. **Preamble** — Load `agent-context.env`, run `linkedout status --json` to verify system
3. **Schema reference** — Key tables: `connection`, `crawled_profile`, `experience`, `education`, `profile_skill`, `company`, `company_alias`, `role_alias`, `funding_round`, `startup_tracking`, `growth_signal`. Column descriptions. Key joins. RLS context (always filter by tenant/BU/user).
4. **Query routing** — Structured lookups (company, location, role, skill) → psql. Semantic search → `linkedout embed` + pgvector `<=>` operator. Complex intelligence → explain reasoning.
5. **Output formatting** — Tables for lists. Summaries for single results. "Why this person" explanations.
6. **Follow-ups** — After every result, suggest 2-3 related queries.

**Files to create:**
- `skills/linkedout/SKILL.md.tmpl` — Core query skill template
- `skills/linkedout/schema-reference.md` — DB schema (auto-generated from `build_schema_context()` in `schema_context.py`, NOT hand-written)
- `bin/generate-schema-ref` — Script that calls `build_schema_context()` and writes schema-reference.md

**Integration points:**
- `docs/decision/env-config-design.md` — `~/linkedout-data/config/agent-context.env` path
- `docs/decision/cli-surface.md` — `linkedout status --json` output format
- Phase 5 embedding abstraction — pgvector query syntax for semantic search

**Testing:**
- Manual: Install skill in Claude Code, run `/linkedout "who do I know at Stripe?"` → should return results
- Manual: Run `/linkedout "find ML engineers in SF"` → should use semantic search
- Verify: Generated SKILL.md is valid for each host

**Complexity:** L

### 8J. Generation Script (`bin/generate-skills`)

**Goal:** A single script that reads all `SKILL.md.tmpl` files, applies host configs, and writes generated `SKILL.md` files to the output directories.

**Acceptance criteria:**
- Script at `bin/generate-skills` (Python, executable)
- Reads all `skills/*/SKILL.md.tmpl` files
- Reads host configs from `skills/hosts/*.yaml`
- Generates output to `skills/claude-code/*/SKILL.md`, `skills/codex/*/SKILL.md`, `skills/copilot/*/SKILL.md`
- Also generates routing files (CLAUDE.md / AGENTS.md)
- Includes `--host` flag to generate for a single host
- Includes `--dry-run` flag to show what would be generated
- Includes `--diff` flag to show changes vs current generated files
- Validates generated output (valid markdown, frontmatter parses)
- Prints summary: "Generated N skills for M hosts"
- Exit code 1 if any validation fails

**Files to create:**
- `bin/generate-skills` — Main generation script (Python)

**Design decisions:**
- **No watch mode** — Generation runs once. Developers edit `.tmpl` files and re-run. No file watchers.
- **Deterministic output** — Same input always produces same output (no timestamps in generated files). This makes diffs clean and CI can verify no drift.
- **CI integration** — `bin/generate-skills --check` verifies generated files match templates (fail if someone edited generated files directly)

**Integration points:**
- `skills/lib/template.py` (8A) — template engine
- `skills/hosts/*.yaml` (8B) — host configs

**Testing:**
- Run `bin/generate-skills` → verify all expected output files exist
- Run `bin/generate-skills --check` → should pass (generated matches templates)
- Modify a `.tmpl` file → `bin/generate-skills --check` should fail
- Run `bin/generate-skills` again → `--check` should pass

**Complexity:** M

### 8K. Dev Workflow

**Goal:** Provide scripts for developers contributing to skills. `bin/dev-setup` symlinks the repo's skill output into the host's skill directory for live editing. `bin/dev-teardown` removes the symlinks. `linkedout-relink` detects and fixes broken symlinks.

**Acceptance criteria:**
- `bin/dev-setup` — Detects installed AI hosts (Claude Code, Codex, Copilot by checking for `~/.claude/`, `~/.agents/`, etc.). Symlinks `skills/<host>/` into the host's global skill directory. Shows what was linked.
- `bin/dev-teardown` — Removes symlinks created by `dev-setup`. Does not delete non-symlink files (safety).
- `linkedout-relink` CLI command — Checks if skill symlinks exist and are valid. Fixes broken symlinks (e.g., after moving the repo). Reports status.
- All scripts are idempotent — safe to re-run
- All scripts report what they did (linked/unlinked/fixed N skills)

**Files to create:**
- `bin/dev-setup` — Dev symlink setup script (Python or bash)
- `bin/dev-teardown` — Dev symlink teardown script
- Add `linkedout-relink` to CLI commands in `pyproject.toml` (or as a `bin/` script)

**Integration points:**
- `docs/decision/cli-surface.md` — `linkedout-relink` is a dev convenience, not in the main CLI help (similar to how gstack handles `gstack-relink`)
- `CONTRIBUTING.md` — Dev workflow documented there

**Testing:**
- Run `bin/dev-setup` → verify symlinks exist at expected locations
- Run `bin/dev-teardown` → verify symlinks removed
- Break a symlink → run `linkedout-relink` → verify fixed

**Complexity:** S

## Implementation Order

The tasks have these dependencies:

```
8A (Template System) ─┐
8B (Host Configs)     ─┤── 8J (Generation Script) depends on both
                       │
8C (Core Skill)       ─┤── Can start in parallel with 8A/8B (template content)
8D-8H (Stub Skills)  ─┤── Can start after 8A/8B are done
                       │
8I (Routing)          ─┘── After 8C-8H define skill names/descriptions
                       
8K (Dev Workflow)     ─── Independent, can run any time after 8J
```

**Recommended execution order:**
1. **Wave 1 (parallel):** 8A + 8B + 8C (template engine, host configs, core skill content)
2. **Wave 2 (parallel):** 8D + 8E + 8F + 8G + 8H (all stub skills — fast, independent)
3. **Wave 3:** 8J (generation script — needs 8A, 8B, and all skill templates)
4. **Wave 4 (parallel):** 8I + 8K (routing + dev workflow — needs generation working)

## Resolved Decisions (2026-04-07, SJ)

1. **Generated files in git:** **Yes — track in git.** Users can install skills directly from clone without running generation. CI check `bin/generate-skills --check` catches drift. Matches gstack pattern.

2. **Copilot skill format:** **Implement based on current docs, flag as "beta."** Near-zero marginal effort. Three platforms at launch > two. Update when format stabilizes.

3. **Schema reference:** **Auto-generate from existing `build_schema_context()`.** The `schema_context.py` file already introspects SQLAlchemy entities at runtime (`_describe_table()` walks `entity_cls.__table__.columns`). Create `bin/generate-schema-ref` as a thin wrapper (~10 lines) that calls `build_schema_context()` and writes to `skills/linkedout/schema-reference.md`. Include in `bin/generate-skills` pipeline.

4. **Skill prefix:** **No toggle. `/linkedout` is the only prefix.** No configurability needed. No consumer for this feature.

5. **Agent tool in skills:** **Core `/linkedout` skill SHOULD use Agent tool for parallel query execution.** Queries take 30s-1min — parallelizing independent lookups (company alias + semantic search + direct SQL) meaningfully cuts response time. Design the core skill to leverage this. Simple utility skills (`/linkedout-history`, `/linkedout-report`, `/linkedout-setup-report`) stay sequential — nothing to parallelize.

6. **Extension setup skill:** **Do NOT create a stub in Phase 8.** `/linkedout-extension-setup` is created in Phase 12 after UX design gate approval. No stubs, no placeholders.
