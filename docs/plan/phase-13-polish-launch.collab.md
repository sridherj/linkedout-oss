## Task Breakdown

### 13A. Multi-Platform Installation CI

**Goal:** GitHub Actions matrix that runs the Phase 9R installation test suite across target platforms and runtimes. This is the definitive "does setup work?" gate before any release.

**Files to create/modify:**
- `.github/workflows/installation-test.yml` — New workflow
- `tests/installation/` — Test scripts (should exist from Phase 9R)

**CI Matrix:**

| Axis | Values |
|------|--------|
| OS | `ubuntu-24.04`, `macos-latest` |
| Python | `3.11`, `3.12`, `3.13` |
| PostgreSQL | `16`, `17` |

Total combinations: 2 OS x 3 Python x 2 PostgreSQL = 12 matrix jobs.

**Workflow design:**
1. **Trigger:** Nightly schedule (`cron: '0 4 * * *'`), manual dispatch, push to `release/*` branches
2. **Setup:** Install PostgreSQL (version-matched), Python (version-matched), clone repo
3. **Run:** Execute `tests/installation/test_fresh_install.sh` — simulates `git clone` -> `/linkedout-setup` -> `linkedout status`
4. **Assert:** Parse the readiness report JSON and verify: zero gaps, expected table counts, CLI commands functional
5. **Artifacts:** Upload readiness reports and setup logs as workflow artifacts on failure

**Integration with Phase 9R test suite:**

| Test | Runs In CI? | Notes |
|------|-------------|-------|
| Fresh install smoke test | Yes (all matrix) | Core gate |
| Prerequisite detection | Yes (all matrix) | Validates OS-specific detection |
| Idempotency test | Yes (ubuntu-24.04 only) | Run setup twice, verify clean second run |
| Partial failure recovery | Yes (ubuntu-24.04 only) | Kill mid-setup, verify recovery |
| Permission tests | Yes (all matrix) | Verify `secrets.yaml` chmod 600, no secrets in logs |
| Degraded environment | Yes (ubuntu-24.04 only) | Mock GitHub down, invalid API key |
| Readiness report assertion | Yes (all matrix) | Every test ends with readiness report check |
| Upgrade path test | No (manual/release only) | Too slow for nightly; runs on release branches |

**Implementation notes:**
- PostgreSQL installation: use `apt-get` on Ubuntu, `brew` on macOS. Pin major version.
- pgvector: install from `apt` on Ubuntu (`postgresql-16-pgvector`), `brew install pgvector` on macOS
- Python: use `actions/setup-python@v5`
- Seed data: use a minimal test seed fixture (not the full 50MB download) to keep CI fast
- API keys: skip OpenAI/Apify — tests use local nomic embeddings only
- Timeout: 30 minutes per job (installation + embeddings on CPU can be slow)

**Acceptance criteria:**
- [ ] CI matrix runs on nightly schedule without manual intervention
- [ ] All 12 matrix combinations pass (or have documented known issues with clear labels)
- [ ] Failed jobs upload readiness reports and logs as artifacts
- [ ] Workflow runs on `release/*` branch pushes
- [ ] Manual dispatch works with customizable matrix parameters

### 13B. End-to-End Flow Testing

**Goal:** Automated test that exercises the complete user journey: clone -> setup -> import -> query -> upgrade -> re-setup. Uses the readiness report as the test oracle.

**Files to create/modify:**
- `tests/installation/test_e2e_flow.sh` — New end-to-end test script
- `tests/installation/test_upgrade_path.sh` — New upgrade path test
- `.github/workflows/installation-test.yml` — Add e2e job (runs after matrix)

**Test flow:**

```
Step 1: Fresh clone + setup
  - git clone (or use checkout)
  - Run /linkedout-setup equivalent (scripts/system-setup.sh + CLI commands)
  - Assert: readiness report has zero gaps
  
Step 2: Import connections
  - linkedout import-connections tests/fixtures/sample-connections.csv
  - Assert: report shows expected import counts
  
Step 3: Generate embeddings  
  - linkedout embed --provider local
  - Assert: embedding coverage = 100% in status

Step 4: Compute affinity
  - linkedout compute-affinity
  - Assert: all connections have affinity scores

Step 5: Query (CLI smoke)
  - linkedout status --json
  - Assert: JSON output has expected fields and non-zero counts
  - linkedout diagnostics --json
  - Assert: all health checks pass

Step 6: Simulate upgrade
  - Modify VERSION file to simulate version bump
  - linkedout migrate --dry-run
  - Assert: no pending migrations (clean state)

Step 7: Re-run setup (idempotency)
  - Re-run setup steps
  - Assert: all steps skip (already complete)
  - Assert: readiness report identical to Step 1
```

**Test fixtures needed:**
- `tests/fixtures/sample-connections.csv` — Small LinkedIn-format CSV (50-100 rows) with realistic data
- `tests/fixtures/sample-seed.sqlite` — Minimal seed data for testing (companies matching the CSV connections)

**Implementation notes:**
- Each step checks the exit code AND parses the report JSON
- On failure, dump `~/linkedout-data/logs/` and the last readiness report
- The e2e test runs on ubuntu-24.04 + Python 3.12 + PostgreSQL 17 only (representative, not exhaustive)
- Runs after the matrix CI, gated on matrix passing

**Acceptance criteria:**
- [ ] Full flow completes in < 15 minutes on CI
- [ ] Each step produces a parseable report artifact
- [ ] Idempotency check (step 7) shows zero data loss and zero gaps
- [ ] Test failures produce enough diagnostic output to debug remotely
- [ ] Sample CSV and seed fixtures are committed and documented

### 13E. Observability Validation

**Goal:** Verify that every subsystem produces the expected logging, metrics, and report artifacts per the logging & observability decision doc. This is a validation task, not a build task — if gaps are found, file issues or fix inline.

**Verification checklist:**

#### Logging infrastructure
- [ ] All backend modules use `get_logger(__name__)` from `shared.utilities.logger` (no raw `import logging`)
- [ ] Every log entry carries `component` and `operation` fields via loguru `bind()`
- [ ] Console output is human-readable (`HH:mm:ss LEVEL module message`)
- [ ] File logs go to `~/linkedout-data/logs/` (not `backend/logs/`)
- [ ] Per-component log files exist: `backend.log`, `cli.log`, `setup.log`, `enrichment.log`, `import.log`, `queries.log`
- [ ] Log rotation: 50MB per file, 30-day retention, gzip compression to `archive/`
- [ ] `LINKEDOUT_LOG_LEVEL` and `LOG_LEVEL_<MODULE>` overrides work

#### Correlation IDs
- [ ] Backend API: every request gets a `correlation_id` (nanoid, prefix `req_`)
- [ ] CLI: every command invocation gets a `correlation_id` (prefix `cli_`)
- [ ] Backend returns `X-Correlation-ID` response header

#### CLI operation result pattern
Verify EVERY CLI command follows the pattern:
```
1. Progress during execution
2. Summary on completion (succeeded/skipped/failed)
3. Failures listed with reasons (if any)
4. Report persisted to ~/linkedout-data/reports/
```

| Command | Progress | Summary | Failures | Report |
|---------|----------|---------|----------|--------|
| `import-connections` | [ ] | [ ] | [ ] | [ ] |
| `import-contacts` | [ ] | [ ] | [ ] | [ ] |
| `compute-affinity` | [ ] | [ ] | [ ] | [ ] |
| `embed` | [ ] | [ ] | [ ] | [ ] |
| `download-seed` | [ ] | [ ] | [ ] | [ ] |
| `import-seed` | [ ] | [ ] | [ ] | [ ] |
| `diagnostics` | N/A | [ ] | [ ] | [ ] |
| `reset-db` | N/A | [ ] | N/A | N/A |

#### Metrics
- [ ] `~/linkedout-data/metrics/daily/YYYY-MM-DD.jsonl` populated after operations
- [ ] `summary.json` updated by `linkedout status`
- [ ] Metric events include: `ts`, `metric`, `value`, and relevant context fields

#### Diagnostic report
- [ ] `linkedout diagnostics` produces all sections: System, Config, Database, Health, Recommendations
- [ ] Secrets are never included (only "configured" / "not configured")
- [ ] `--json` flag outputs valid JSON
- [ ] `--repair` flag detects and offers to fix common issues
- [ ] Report is saved to `~/linkedout-data/reports/diagnostic-YYYYMMDD-HHMMSS.json`

#### Readiness reports
- [ ] Every major operation (`import-connections`, `compute-affinity`, `embed`, `import-seed`, setup flow) produces a report in `~/linkedout-data/reports/`
- [ ] Reports follow the standard format: `operation`, `timestamp`, `duration_ms`, `counts`, `coverage_gaps`, `failures`, `next_steps`
- [ ] Setup readiness report includes precise numbers (not just pass/fail)

**How to validate:** Run the full user flow (import -> embed -> affinity -> diagnostics) on a dev machine and inspect all output files. Create a validation script at `scripts/validate-observability.sh` that runs the flow and checks for expected artifacts.

**Files to create:**
- `scripts/validate-observability.sh` — Automated observability validation script

**Acceptance criteria:**
- [ ] Every checkbox above is verified
- [ ] Gaps found are documented as GitHub issues with clear descriptions
- [ ] `scripts/validate-observability.sh` passes cleanly on a working installation
- [ ] No silent failures — every operation that modifies data reports what it did

## Open Questions

1. **Windows/WSL testing in CI:** The high-level plan lists Windows as "community-supported" with WSL recommended. Should the CI matrix include a `windows-latest` runner with WSL2? This would significantly increase CI complexity and cost. **Recommendation:** Defer Windows CI to post-v1. Document WSL instructions in the getting-started guide but don't gate the release on Windows CI passing.

2. **Seed data hosting:** The release plan assumes seed data files (50MB + 500MB) are uploaded as GitHub Release assets. GitHub has a 2GB per-asset limit, so this works. But should we also host on a CDN or mirror for faster downloads in non-US regions? **Recommendation:** GitHub Releases for v1. Add CDN mirror if download speed becomes a reported issue.

3. **LLM eval tier scope:** The optional Tier 4 (LLM eval) is listed but not detailed. What queries should it test? What's the pass/fail criteria for "good results"? **Recommendation:** Define a small eval set (20-30 queries with expected result patterns) but don't gate v0.1.0 on it. This is informational quality tracking, not a release gate.

4. **Announcement channel:** Where should v0.1.0 be announced? Hacker News? Reddit r/selfhosted? LinkedIn? **Recommendation:** Decide closer to launch. The release workflow should NOT auto-announce — SJ should control timing and messaging.

5. **Pre-built extension versioning:** The extension zip is built from the same repo. Should it have its own version number, or track the repo version? **Recommendation:** Track repo version. Extension manifest version = repo `VERSION` file. Simpler for users to reason about.
