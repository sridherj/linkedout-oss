## Objective

Build the two query tools (`execute_sql`, `search_profiles`) and the dynamic schema context builder. These are the foundational pieces that the SearchAgent (SP-3) will call via LLM tool-calling. Also incorporate unenriched connection handling into the schema context so the LLM generates correct queries for both enriched and stub profiles.

## Context

The tools are ported from `<prior-project>/agents/linkedin/tools.py` but adapted for:
1. LinkedOut's SQLAlchemy-based DB layer (not raw psycopg)
2. User-scoped queries via `app_user_id` (JOIN through `connection` table)
3. New schema: `crawled_profile` + `connection` (not old `profiles` + `affinity_scores`)
4. Stub `crawled_profile` rows for unenriched connections (`has_enriched_data=FALSE`)

## Step 1: Schema Context Builder (`schema_context.py`)

**What:** Dynamically generates the schema reference that goes into the SearchAgent's system prompt. Tells the LLM what tables/columns exist and how to query them.

**Implementation:**

```python
def build_schema_context(session: Session) -> str:
    """Build schema reference string from entity metadata for LLM system prompt."""
```

**Must include:**
- Table names, column names, types, and comments from `CrawledProfileEntity`, `ConnectionEntity`, `ExperienceEntity`, `EducationEntity`, `CompanyEntity`
- Business rules:
  - "connection table is user-scoped via app_user_id"
  - "crawled_profile is shared; always JOIN through connection for user's network"
  - "All connections have a crawled_profile (stub or enriched). Check `cp.has_enriched_data` to distinguish."
  - "Stub profiles (has_enriched_data=FALSE) have basic CSV data (name, company, title) but no embedding, no about, no experience/education. They cannot be found via vector search."
  - "Always use regular JOIN, never LEFT JOIN."
  - Available `dunbar_tier` values: `inner_circle`, `active`, `familiar`, `acquaintance`
  - Affinity signal columns on connection table
- Example user-scoping JOIN pattern:
  ```sql
  FROM crawled_profile cp
  JOIN connection c ON c.crawled_profile_id = cp.id
  WHERE c.app_user_id = :app_user_id
  ```

## Step 2: SQL Tool (`sql_tool.py`)

**What:** Executes structured SQL queries generated by the LLM. User-scoped, SELECT-only, with safety guardrails.

**Porting from:** `<prior-project>/agents/linkedin/tools.py` → `execute_sql()`

**Key changes from old code:**

| Aspect | Old (second-brain) | New (LinkedOut) |
|--------|-------------------|-----------------|
| Connection | `psycopg.connect(get_dsn())` | `DbSessionManager` session + `session.execute(text(query))` |
| User scoping | Not enforced | `app_user_id` injected as bind parameter |
| Tables | `profiles`, `affinity_scores` | `crawled_profile`, `connection`, `experience`, `education`, `company` |
| Safety | SELECT-only, LIMIT inject | Same + validate `app_user_id` in WHERE clause post-parse |
| Error hints | Old table/column names | Updated for LinkedOut schema |

**Implementation details:**

1. **SELECT-only validation:** Parse query, reject anything that's not SELECT
2. **LIMIT injection:** `_inject_limit(query, max_rows=100)` — add LIMIT if missing
3. **Statement timeout:** 5 second timeout on query execution
4. **Parameter injection:** Tool receives `app_user_id` from SearchAgent. Replace `:app_user_id` placeholder with a bind parameter — never string-interpolate
5. **Post-execution validation:** Warn (log) if generated SQL doesn't contain `app_user_id` binding — this is a safety net, not a hard block (some aggregate queries across user's network may not need it in the WHERE if they already scope through a subquery)
6. **Error hints:** On column-not-found errors, include available columns for that table. On table-not-found, list available tables.
7. **Return format:** `{"columns": [...], "rows": [...], "row_count": N}` — same as old code

**Function signature:**

```python
def execute_sql(query: str, app_user_id: str, session: Session) -> dict:
    """Execute a user-scoped SQL query. Returns {columns, rows, row_count}."""
```

## Step 3: Vector Tool (`vector_tool.py`)

**What:** Semantic search over user's network using pgvector cosine similarity.

**Porting from:** `<prior-project>/agents/linkedin/tools.py` → `search_profiles()`

**Key changes from old code:**

| Aspect | Old (second-brain) | New (LinkedOut) |
|--------|-------------------|-----------------|
| User scoping | None | JOIN through connection WHERE app_user_id |
| Tables | `profiles` | `crawled_profile` JOIN `connection` |
| Columns | `p.first_name, p.last_name` | `cp.full_name` |
| Affinity | `affinity_scores.total_score` | `connection.affinity_score` |
| Embeddings | `LLMClient.get_embeddings()` | LinkedOut's `EmbeddingClient` |
| pgvector | `p.embedding <=> %s::vector` | Same operator, SQLAlchemy `text()` with bind params |

**Implementation details:**

1. **Embed the query:** Use `EmbeddingClient` to generate embedding for the search query
2. **User-scoped vector search SQL:**
   ```sql
   SELECT cp.id, cp.full_name, cp.headline, cp.current_position,
          cp.current_company_name, cp.location_city, cp.location_country,
          cp.linkedin_url, cp.public_identifier,
          c.id as connection_id, c.affinity_score, c.dunbar_tier, c.connected_at,
          cp.has_enriched_data,
          1 - (cp.embedding <=> :query_embedding::vector) AS similarity
   FROM crawled_profile cp
   JOIN connection c ON c.crawled_profile_id = cp.id
   WHERE c.app_user_id = :app_user_id
     AND cp.embedding IS NOT NULL
   ORDER BY cp.embedding <=> :query_embedding
   LIMIT :limit
   ```
3. **Only returns enriched profiles** (stub profiles have no embedding, so `embedding IS NOT NULL` naturally excludes them)
4. **Return format:** List of dicts with profile data + similarity score

**Function signature:**

```python
def search_profiles(query: str, app_user_id: str, session: Session, limit: int = 20) -> list[dict]:
    """Semantic search over user's network. Returns ranked results with similarity scores."""
```
