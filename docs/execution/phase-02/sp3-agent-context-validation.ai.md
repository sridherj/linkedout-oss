# SP3: Agent Context File + Startup Validation

**Sub-phase:** 3 of 7
**Tasks covered:** 2E (Agent context), 2F (Startup validation)
**Size:** S + M
**Dependencies:** SP2 (config module must exist)
**Estimated effort:** 30-45 minutes

## Objective

Add the agent context file generator and startup validation with actionable error messages to the config system built in SP2.

---

## Steps

### 1. Create `backend/src/shared/config/agent_context.py`

**Function:** `generate_agent_context(settings: LinkedOutSettings) -> Path`

**Implementation:**
- Takes a `LinkedOutSettings` instance
- Writes to `{settings.data_dir}/config/agent-context.env`
- Content format:
  ```bash
  # Auto-generated by LinkedOut setup — do not edit manually
  # Source this file in CLAUDE.md or skill definitions for DB access
  DATABASE_URL=<from settings.database_url>
  LINKEDOUT_TENANT_ID=tenant_sys_001
  LINKEDOUT_BU_ID=bu_sys_001
  LINKEDOUT_USER_ID=usr_sys_001
  ```
- System tenant/BU/user IDs are hardcoded constants (same as in `extension/lib/constants.ts`)
- Overwrites existing file on each call (idempotent)
- Returns the `Path` to the written file
- Ensures parent directory exists (`config/` under data_dir)

**Note:** This function is called by `/linkedout-setup` (Phase 9), not during normal startup. It's defined here because it's config infrastructure.

### 2. Add Startup Validators to `backend/src/shared/config/settings.py`

Add pydantic `@field_validator` decorators for:

**2a. `database_url` — Required field with actionable error:**
- The field has a default value (`postgresql://linkedout:@localhost:5432/linkedout`), so it won't be "missing" in the pydantic sense
- Add a validator that checks the URL format (starts with `postgresql://` or `postgres://`)
- If malformed, show an actionable error explaining the three ways to set it

**2b. `openai_api_key` — Conditionally required:**
- Add a `@model_validator(mode='after')` that checks: if `embedding_provider == 'openai'` and `openai_api_key is None`, raise a validation error with message:
  ```
  OPENAI_API_KEY is required when embedding_provider is 'openai'.
  
  Set it in one of:
    1. ~/linkedout-data/config/secrets.yaml  →  openai_api_key: sk-...
    2. Environment variable                  →  export OPENAI_API_KEY=sk-...
  
  Or switch to local embeddings: LINKEDOUT_EMBEDDING_PROVIDER=local
  ```

**2c. `backend_port` — Range validation:**
- `@field_validator('backend_port')` — must be 1-65535
- Error: `"backend_port must be between 1 and 65535, got {value}"`

**2d. `log_level` — Enum validation:**
- `@field_validator('log_level')` — must be one of `DEBUG`, `INFO`, `WARNING`, `ERROR`
- Case-insensitive (normalize to uppercase)
- Error: `"log_level must be one of: DEBUG, INFO, WARNING, ERROR. Got: '{value}'"`

**2e. `log_format` — Enum validation:**
- `@field_validator('log_format')` — must be one of `human`, `json`
- Error: `"log_format must be one of: human, json. Got: '{value}'"`

### 3. Custom Error Formatting

Override pydantic's default `ValidationError` display to produce the actionable multi-line error format from the decision doc. This can be done via a wrapper in `get_config()`:

```python
def get_config() -> LinkedOutSettings:
    global _settings_instance
    if _settings_instance is None:
        try:
            _settings_instance = LinkedOutSettings()
        except ValidationError as e:
            # Format as actionable error message
            print("ERROR: LinkedOut configuration invalid.\n", file=sys.stderr)
            for error in e.errors():
                field = error['loc'][0] if error['loc'] else 'unknown'
                msg = error['msg']
                print(f"  {field}: {msg}\n", file=sys.stderr)
            sys.exit(1)
    return _settings_instance
```

- [ ] Invalid port number (outside 1-65535) produces clear error
- [ ] Invalid log level produces error with valid options listed
- [ ] Invalid log format produces error with valid options listed
- [ ] Log level validation is case-insensitive
