# Sub-Phase 4: Generation Script (`bin/generate-skills`)

**Phase:** 8 — Skill System & Distribution
**Plan tasks:** 8J (Generation Script)
**Dependencies:** sp1 (Template Engine + Host Configs), sp2 (Core Query Skill), sp3 (Stub Skills)
**Blocks:** sp5
**Can run in parallel with:** —

## Objective
Build the `bin/generate-skills` script that reads all `SKILL.md.tmpl` files, applies host configs, and writes generated `SKILL.md` files to per-host output directories. This is the "build system" for skills — it must be deterministic, validating, and CI-friendly.

## Context
- Read shared context: `docs/execution/phase-08/_shared_context.md`
- Read plan (8J section): `docs/plan/phase-08-skill-system.md`
- Read template engine code from sp1: `skills/lib/template.py`, `skills/lib/config.py`, `skills/lib/frontmatter.py`
- Read all skill templates from sp2 and sp3: `skills/*/SKILL.md.tmpl`
- Read host configs from sp1: `skills/hosts/*.yaml`

## Deliverables

### 1. `bin/generate-skills` (NEW)

Main generation script. Python, executable (`#!/usr/bin/env python3`).

**Core behavior:**
1. Discover all `skills/*/SKILL.md.tmpl` files
2. Discover all host configs from `skills/hosts/*.yaml`
3. For each template × host combination:
   a. Load template
   b. Load host config
   c. Build variable context (host vars + global context)
   d. Render template (resolve variables, apply conditionals)
   e. Process frontmatter (apply host rules: denylist/allowlist, field stripping, description limits)
   f. Apply path rewrites in rendered content
   g. Apply tool name rewrites in rendered content
   h. Add generated header: `<!-- Generated by bin/generate-skills — do not edit -->`
   i. Write to output path: `skills/{host-dir}/{skill-name}/SKILL.md`
4. Also call `bin/generate-schema-ref` to update schema reference before rendering the core skill
5. Print summary: `Generated N skills for M hosts`

**Host output directory mapping:**
- `claude` → `skills/claude-code/`
- `codex` → `skills/codex/`
- `copilot` → `skills/copilot/`

**CLI flags:**
- `--host HOST` — Generate for a single host only (e.g., `--host claude`)
- `--dry-run` — Show what would be generated without writing files
- `--diff` — Show changes vs current generated files (like `git diff` but without git)
- `--check` — Verify generated files match templates. Exit 0 if match, exit 1 if drift detected. For CI use.

**Validation (run after generation):**
- All generated files are valid markdown
- All generated files have parseable YAML frontmatter
- No unresolved `{{variables}}` in any generated file
- Exit code 1 if any validation fails, with clear error messages

**Deterministic output:**
- Same input always produces same output
- No timestamps, random values, or environment-dependent content in generated files
- This ensures `--check` is reliable and diffs are clean

### 2. Integration with `bin/generate-schema-ref`

The generation pipeline should:
1. First run `bin/generate-schema-ref` (updates `skills/linkedout/schema-reference.md`)
2. Then render all templates (which may reference the schema reference content)

If `bin/generate-schema-ref` fails (e.g., no DB available), continue with the existing `schema-reference.md` file and print a warning.

### 3. Unit/Integration Tests

**`tests/skills/test_generation.py`** (NEW)

End-to-end generation tests:
- `test_generate_all_skills`: Run full generation, verify all expected output files exist
- `test_generate_single_host`: Run with `--host claude`, verify only Claude output generated
- `test_check_passes_after_generation`: Generate → `--check` → exit 0
- `test_check_fails_on_drift`: Generate → modify a generated file → `--check` → exit 1
- `test_no_unresolved_variables`: After generation, grep all output for `{{` → should find zero matches
- `test_generated_header_present`: All generated files start with the `<!-- Generated by ... -->` header
- `test_frontmatter_valid`: All generated files have parseable YAML frontmatter
- `test_deterministic_output`: Generate twice → output is byte-identical

## Verification
1. `python bin/generate-skills` succeeds and prints summary
2. All 18 expected output files exist
3. `python bin/generate-skills --check` exits 0
4. Modify any generated file → `python bin/generate-skills --check` exits 1
5. `python bin/generate-skills --dry-run` prints plan without writing files
6. `python bin/generate-skills --host claude` only generates Claude Code files
7. `grep -r '{{' skills/claude-code/ skills/codex/ skills/copilot/` returns no matches
8. `pytest tests/skills/test_generation.py -v` passes

## Notes
- The script should be fast — generating 18 markdown files should take < 1 second.
- `--check` mode is critical for CI. It prevents drift between templates and generated output. Someone who edits a generated file directly will get caught by CI.
- The `--diff` flag is a developer convenience — helps see what changed before committing. Use Python's `difflib` for this, not shell `diff`.
- Generated files ARE committed to git (per decision #1). This means users can install skills directly from a clone without running generation.
- No watch mode. Developers edit `.tmpl` files and re-run. No file watchers.
