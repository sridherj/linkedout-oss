# Sub-Phase 2: Core Query Skill (`/linkedout`)

**Phase:** 8 — Skill System & Distribution
**Plan tasks:** 8C (/linkedout Skill)
**Dependencies:** sp1 (Template Engine + Host Configs)
**Blocks:** sp4
**Can run in parallel with:** sp3

## Objective
Create the flagship `/linkedout` skill — the primary user interface. Users invoke `/linkedout` with natural language queries about their professional network. The skill routes to psql queries or CLI commands, formats results, and suggests follow-ups. This is the largest single deliverable in Phase 8.

## Deliverables

### 1. `bin/generate-schema-ref` (NEW)

Thin wrapper script (~10-15 lines) that:
- Calls `build_schema_context()` from `backend/src/utilities/llm_manager/schema_context.py`
- Writes the output to `skills/linkedout/schema-reference.md`
- Adds a header: `<!-- Auto-generated by bin/generate-schema-ref — do not edit -->`
- Executable: `#!/usr/bin/env python3`

Requirements:
- Must be runnable standalone: `python bin/generate-schema-ref`
- Also callable from `bin/generate-skills` as part of the generation pipeline
- If `build_schema_context()` requires a DB connection, make the script handle the missing-DB case gracefully (skip generation, log a warning, exit 0) — the schema reference is checked into git so a stale version is acceptable when DB is unavailable

### 2. `skills/linkedout/schema-reference.md` (NEW — auto-generated)

Generated by `bin/generate-schema-ref`. Contains:
- DB schema for key tables: `connection`, `crawled_profile`, `experience`, `education`, `profile_skill`, `company`, `company_alias`, `role_alias`, `funding_round`, `startup_tracking`, `growth_signal`
- Column names, types, and descriptions
- Key relationships/joins
- RLS context: always filter by tenant_id, business_unit_id, user_id

If `build_schema_context()` cannot run (no DB), create a hand-written initial version based on the entity definitions in `backend/src/linkedout/*/entities/`. This will be overwritten by the auto-generator when a DB is available.

### 3. `skills/linkedout/SKILL.md.tmpl` (NEW)

The core query skill template. Structure:

**Frontmatter:**
```yaml
---
name: linkedout
description: Query your professional network — find connections, companies, roles, and insights from your LinkedIn data
tools:
  - Bash
  - Read
  - Grep
  - Agent
---
```

**Body sections (in order):**

1. **Preamble**
   - Load `{{AGENT_CONTEXT_PATH}}` to get DB credentials and tenant/BU/user IDs
   - Run `{{CLI_PREFIX}} status --json` to verify system health
   - If status check fails, explain what's wrong and suggest `{{CLI_PREFIX}} diagnostics`

2. **Schema Reference**
   - Include contents of `schema-reference.md` (or reference it via `{{DB_SCHEMA_SNIPPET}}` variable)
   - Key tables, columns, relationships
   - RLS context: always include `WHERE tenant_id = $TENANT_ID AND business_unit_id = $BU_ID AND user_id = $USER_ID`

3. **Query Routing**
   - **Structured lookups** (company, location, role, skill) → direct SQL via `psql` or `{{CLI_PREFIX}}` commands
   - **Semantic search** ("people like...", "similar to...") → `{{CLI_PREFIX}} embed` + pgvector `<=>` operator with `embedding_nomic vector(768)` or `embedding_openai vector(1536)` depending on configured provider
   - **Complex intelligence** ("who should I reach out to for...", "warm intros to...") → combine structured + semantic, explain reasoning
   - Include SQL patterns for common queries:
     - Find connections at a company (with company_alias join)
     - Find people by role (with role_alias join)
     - Find people by location
     - Find people by skill
     - Semantic similarity search
     - Funding-aware queries (join funding_round, startup_tracking)

4. **Parallel Query Execution**
   - For complex queries, use the Agent tool to run independent lookups in parallel
   - Example: company alias lookup + semantic search + direct SQL can run concurrently
   - Only for the core `/linkedout` skill — per decision #5

5. **Output Formatting**
   - Tables for list results (name, company, role, location)
   - Rich summaries for single-person results
   - "Why this person" explanations for semantic/intelligence queries
   - Always show result count

6. **Follow-ups**
   - After every result, suggest 2-3 related queries
   - Examples: "Want to see their career history?", "Find more people at [company]?", "See funding data for [company]?"

**Host-specific sections:**
- Use `{{#if HOST_NAME == "claude"}}` for Claude Code-specific content (e.g., Agent tool usage)
- Use `{{#if HOST_NAME == "codex"}}` for Codex-specific tool references
- Path references use `{{DATA_DIR}}`, `{{CONFIG_DIR}}`, etc.

## Notes
- The skill template is a large markdown file (likely 200-400 lines). This is expected — it needs to contain enough context for Claude/Codex/Copilot to write correct SQL queries.
- `build_schema_context()` in `schema_context.py` already does the heavy lifting — it walks SQLAlchemy entity classes and produces schema text. `bin/generate-schema-ref` is just a thin CLI wrapper.
- The schema reference is intentionally verbose — Claude needs column names, types, and relationship hints to write correct JOINs.
- SQL patterns should use `$TENANT_ID`, `$BU_ID`, `$USER_ID` as placeholders — the skill preamble resolves these from `agent-context.env`.
- The Agent tool for parallel queries is Claude Code-specific. Wrap that section in `{{#if HOST_NAME == "claude"}}`.
